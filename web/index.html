The turn loop

Two phases, and silence is a first-class answer.

Every turn, a separate memory agent reads the trajectory. It never touches your code and never writes into the conversation unless it has a reason it can cite.

PHASE 1

Observe & write

The memory agent updates its bank from the turn: requirements it saw, environment facts, attempts that failed, diagnoses that stuck. Runs on every turn. Costs one small model call.

always runs
PHASE 2 · A

Stay silent

The agent is on track, or the fact is already visible on screen. Adding a reminder would only add noise. Nothing is injected. This is the default, and most turns end here.

the common case
PHASE 2 · B

Speak

The next action is about to break a stated requirement, repeat a stored failure, or ignore a diagnosis in the bank. One reminder, under 120 tokens, citing entry ids you can look up.

only with evidence

The bank

Every reminder points at a row you can read.

No opaque embeddings deciding your session. The bank is a short list of typed entries, each capped at 60 tokens, each with an id. Run agentmem bank to see exactly what your agent is carrying.

ENTRYCONTENTSALIENCE
K-002 Files under generated/ and tests/fixtures/ are derived from schema/*.yaml; run make generate after any schema edit. requirement
P-014 Adding display_name to the serializer raised KeyError; fixtures on disk predate the schema change. diagnosis
K-005 Test suite runs with uv run pytest; plain pytest misses the project venv. environment
P-006 Tried pip install -e . in session 44; fails behind the corporate proxy. dormant

Typed, not vibed. K for facts the session established, P for what was tried and what it cost, plus a running status of the task.

Entries decay. What never gets used fades to dormant and stops competing for context. What keeps changing the agent's behaviour for the better stays sharp.

You can audit any turn. agentmem replay shows what the memory agent saw, what it wrote, and why it stayed quiet.

Evaluation

A memory layer that talks too much is worse than none.

So the harness measures restraint as strictly as recall. Below: the latest LongRun-sim run, 30 interleaved sessions across three repos, requirements stated early, known failures resurfacing late. The harness ships in the repo; one command reproduces it.

Retention · session 30 0% vs 78%

Requirements still steering the agent

Without memory, none of the early requirements survive to the end of the run. With AgentMem, 9 of 9 probed facts are still surfaced by the bank at session 30.

Recurring failures 3 of 3 caught

Reminded before the repeat

Each repo's known failure comes back late in the run. The reminder fired on every recurrence, and only 4 times across the 27 routine turns. Silence stays the default.

Bank growth 1.08x

Thirty sessions, no bloat

Salience decay, consolidation, and promotion keep the bank a short read instead of a scrapbook. Cited cross-repo leaks on one shared bank: zero.

Measured July 2026 with claude-haiku-4-5 as the memory model, about $0.30 of API spend for the full 30-session run. The same harness exposed a retention gap at 56% and drove the lifecycle fixes that closed it to 78%. Reproduce it: evals/longrun_sim/ in the repo.
